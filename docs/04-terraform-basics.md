# 04 - Terraform Basics

## What is Terraform?

Terraform is a tool that lets you describe your cloud infrastructure in text files,
and then creates/updates/deletes it for you.

**Without Terraform**: Click around AWS Console to create stuff manually.
Hard to repeat, easy to forget what you did, no history of changes.

**With Terraform**: Write a text file describing what you want → Terraform creates it.
Want to delete everything? One command. Want the same setup again? One command.

This is called **Infrastructure as Code (IaC)** - your infrastructure is described
in code files that you can version control with git, review in PRs, and reuse.

## The 4 Commands You'll Use 90% of the Time

| Command              | What it does                                          | Costs money? |
| -------------------- | ----------------------------------------------------- | ------------ |
| `terraform init`     | Downloads plugins (like `npm install`)                | No           |
| `terraform plan`     | Preview what will change (dry run, no changes made)   | No           |
| `terraform apply`    | Actually create/change resources in AWS               | **Possibly** |
| `terraform destroy`  | Delete everything Terraform created                   | No           |

### terraform init

- Downloads the provider plugins listed in `providers.tf` (e.g., the AWS plugin)
- Creates a `.terraform/` folder (like `node_modules/`) with the downloaded plugins
- Creates `.terraform.lock.hcl` (like `package-lock.json`) to pin exact versions
- **Local only** - does NOT talk to AWS, does NOT create resources, does NOT cost money
- Run once per project, or after changing provider versions

### terraform plan

- Reads your `.tf` files and compares them to what currently exists in AWS
- Shows you a preview: "I will create X, change Y, delete Z"
- **Read only** - does NOT make any changes
- Always run this before `apply` to verify what's about to happen

### terraform apply

- Same as `plan`, but then asks "Do you want to do this?" (yes/no)
- If you say yes, it creates/changes/deletes resources in AWS
- **This is the one that can cost money** - it actually creates real resources
- After it finishes, it saves what it created in a `terraform.tfstate` file

### terraform destroy

- Deletes everything that Terraform created (reads from tfstate to know what to delete)
- Also asks for confirmation before doing anything
- **Always run this when done experimenting** to avoid surprise charges

## Project File Structure

```text
terraform/
├── providers.tf    # Which cloud + credentials (the "who" and "where")
├── variables.tf    # Input values / parameters (the "settings")
├── outputs.tf      # Values to display after apply (the "results")
├── main.tf         # The actual resources to create (added later)
├── .terraform/     # Downloaded plugins (gitignored, like node_modules)
└── terraform.tfstate  # Tracks what was created (gitignored, contains sensitive info)
```

### providers.tf - Which cloud to talk to

```hcl
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws" # Download AWS plugin from HashiCorp registry
      version = "~> 5.0"        # Use any version 5.x (~> means "compatible with")
    }
  }
  required_version = ">= 1.0"   # Minimum Terraform CLI version
}

provider "aws" {
  region  = "us-east-1"        # Which AWS region to create resources in
  profile = "terraform-admin"  # Which AWS CLI profile to authenticate with
}
```

### variables.tf - Reusable settings

```hcl
variable "project_name" {
  description = "A name to tag all resources with"
  type        = string       # Can be: string, number, bool, list, map
  default     = "airflow-mwaa"
}
```

Use in other files with `var.project_name`.

### outputs.tf - Display results after apply

```hcl
output "bucket_name" {
  description = "The name of the S3 bucket"
  value       = aws_s3_bucket.example.id
}
```

After `terraform apply`, this prints the bucket name to your terminal.

## .gitignore for Terraform

```text
**/.terraform/        # Downloaded plugins (large, re-downloadable)
*.tfstate             # State files (contain sensitive resource info)
*.tfstate.backup      # State backups
*.tfvars              # Variable files (may contain secrets)
.terraform.lock.hcl   # Lock file (regenerated by terraform init)
```

## What is Terraform State? (terraform.tfstate)

After `terraform apply` creates resources, it saves a record of what it created
in a file called `terraform.tfstate`. This is how Terraform knows:

- What exists in AWS (so `plan` can compare "desired" vs "actual")
- What to delete when you run `destroy`
- What changed since last time

**Never edit this file manually.** Never commit it to git (it may contain secrets).
Later, we'll store it remotely in S3 instead of locally.
